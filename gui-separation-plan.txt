GUI/Processing Separation Plan for Siril
=========================================
Generated: 2026-04-29

BACKGROUND & FINDINGS
======================

Goal: Separate GTK-specific code from processing logic so that the codebase
can later adopt GTK4 (or another toolkit) by swapping only the GUI layer,
without touching algorithms, I/O, registration, stacking, or filters.

GUIDING PATTERNS
----------------

Two canonical patterns govern this work:

Pattern A – Filter / algorithm split (epf.c model)
  src/filters/X.c  – pure processing: data types, apply functions, sequence
                     wrappers. No GTK, no gui/ includes.
  src/gui/X.c      – all GTK callbacks, signal handlers, static widget
                     pointers, preview logic. Calls into filters/X.c for the
                     actual work.
  This is already done for epf.c; every filter and algorithm file that mixes
  processing and GUI code should be brought to this state.

Pattern B – Widget initialisation (init_statics model)
  Within every src/gui/X.c file, widget pointers are held as static typed
  variables at file scope (e.g. static GtkSpinButton *spin_foo = NULL).
  A single setup_X_widgets() function (or X_dialog_init_statics() following
  the existing convention) populates them via gtk_builder_get_object() and
  is called once from the dialog's "show" signal handler. All other functions
  in the file use the cached pointer directly.
  lookup_widget() is not called outside src/gui/ at all; within src/gui/ it
  is replaced by the init_statics pattern wherever that has not yet been done.

Current state (from callgraph.db analysis):

  - ~70 non-GUI source files contain at least one cross-boundary call

    translation unit that includes it (i.e. almost the entire codebase)
  - Most-called GUI symbols from non-GUI code:

INTERFACE GROUPS FOR THE RESIDUAL ABSTRACTION LAYER
=====================================================

After applying Pattern A the number of cross-boundary calls drops
substantially. The calls that remain in core/, io/, stacking/, and
registration/ (which cannot simply be split because they are not dialog/
filter hybrids) need a thin abstraction layer. These divide into:

  A. Progress / Status
       set_progress_bar_data, set_cursor_waiting
       → gui_iface: report_progress(fraction, message), set_busy(bool)

  B. Logging
       gui_log_message
       → already bridged via siril_log.c; decouple the hard call

  C. Modal dialogs
       siril_message_dialog, siril_confirm_dialog, siril_open_dialog,
       siril_close_dialog, queue_*_message_dialog, siril_data_dialog
       → gui_iface: message(), confirm(), open_named(), close_named()

  D. Image display / redraw
       redraw, queue_redraw, queue_redraw_and_wait_for_it, remap, remap_all,
       invalidate_gfit_histogram, refresh_histogram_if_visible
       → gui_iface: invalidate(), redraw_async(), redraw_sync()

  E. Sequence / image state notifications
       update_sequences_list, fill_sequence_list, clear_sequence_list,
       sequence_list_change_current, display_filename, adjust_reginfo,
       adjust_refimage, update_reg_interface, update_stack_interface,
       update_prepro_interface, set_layers_for_registration, adjust_sellabel,
       set_cutoff_sliders_*, set_display_mode, sliders_mode_set_state,
       initialize_display_mode, update_gfit_histogram_if_needed, etc.
       → gui_iface: on_sequence_opened(), on_sequence_changed(),
                    on_image_loaded(), on_image_closed()

  F. Tab / window switching
       control_window_switch_to_tab, show_or_hide_mask_tab,
       toggle_histogram_window_visibility, etc.
       → gui_iface: switch_tab(), show_panel()

  G. Misc GUI state
       set_GUI_CAMERA, set_GUI_DiskSpace, set_GUI_MEM,
       update_display_fwhm, update_zoom_label, update_spinCPU,
       script_widgets_enable, queue_activate_action_if_enabled
       → gui_iface: update_status_bar(), update_menu_state()

    -- What does a function call?
    SELECT callee_name, call_line FROM calls
      JOIN functions ON calls.caller_id = functions.id
      WHERE functions.name = 'my_func' ORDER BY call_line;

    -- What calls a function?
    SELECT f.name, f.file, f.line FROM calls
      JOIN functions f ON calls.caller_id = f.id
      WHERE calls.callee_name = 'my_func';

    -- What global vars does a function use?
    SELECT var_name, var_type, use_line FROM var_uses
      JOIN functions ON var_uses.function_id = functions.id
      WHERE functions.name = 'my_func' ORDER BY use_line;

    -- All functions in a file:
    SELECT name, line FROM functions WHERE file LIKE '%filename%' ORDER BY line;

    -- Functions that use a specific global:
    SELECT DISTINCT f.name, f.file FROM var_uses vu
      JOIN functions f ON vu.function_id = f.id
      WHERE vu.var_name = 'com' ORDER BY f.name;
